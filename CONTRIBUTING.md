# Contributing to Solo

Thanks for contributing.

## Reporting Bugs

Use the GitHub issue templates.

Bug reports should be written for verification, not persuasion.

### Writing Standard

Solo issue reports follow a plain, evidence-first writing style inspired by Wikipedia's guidance on neutral point of view, words to watch, and no original research.

- Use clear, direct language.
- State observed facts before theories.
- Separate evidence from interpretation.
- Use exact commands, versions, timestamps, logs, paths, and error text.
- Attribute claims to a source when the source matters.
- Avoid vague scale words like `randomly`, `often`, `huge`, `broken`, or `sometimes` unless you quantify them.
- Avoid loaded or promotional wording like `obviously`, `clearly`, `massive`, `game-changing`, `unacceptable`, or `best-in-class`.
- Do not combine separate observations into a conclusion unless the conclusion is directly supported by the evidence you included.

### No Filler

Do not submit filler text.

Every sentence in an issue should be one of:

- a directly observed fact
- a reproducible step
- a clearly labeled hypothesis

Remove generic summaries, padded prose, and unsupported claims such as:

- `clearly`
- `obviously`
- `widely`
- `randomly`
- `some users report`
- `research shows`

Issues may be closed until they include concrete, actionable evidence.

### Minimum Bug Report Quality

A good bug report includes:

- Summary
- Environment
- Steps to reproduce
- Expected behavior
- Actual behavior
- Evidence
- Scope or impact
- Hypothesis, if any

## Reporting Feature Requests

Feature requests should describe:

- the problem
- who is affected
- the smallest useful behavior change
- any alternatives already considered

Keep requests concrete. Avoid roadmap speculation and marketing language.
